# nito media 記事の書き方ガイド

このガイドは人間・Claude共通のルールです。記事を書く際は必ずここに従ってください。

## 仕組み

1. `media/posts/` に `記事slug.md` を追加して main にpush（GitHubのWeb UIからアップロードでもOK）
2. GitHub Actions が自動で HTML に変換し、`nito-0210.com/media/記事slug/` に公開される
3. 一覧ページ `nito-0210.com/media/` も自動更新される

**`media/posts/` 以外のファイルは手で編集しないこと**（`media/<slug>/index.html` や `media/index.html` は自動生成なので直接編集しても次のビルドで消える）。

## ファイル名（slug）のルール

- 英小文字・ハイフンのみ。URLになるので簡潔に内容を表す
- 例: `aeo-basics.md` → `nito-0210.com/media/aeo-basics/`

## mdファイルの形式

```markdown
---
title: 記事タイトル（32字前後・検索意図に正面から答える）
date: 2026-07-06
category: AI検索対策
description: メタディスクリプション。100〜120字で記事の要点と読むメリットを書く。
image: /media/images/aeo-basics.png
---

導入文（リード）。最初の「## 見出し」より前に書いたものがリードになる。
タイトルの問いに1〜2文で即答する（結論ファースト）。長い前置きは書かない。

:::takeaways
- この記事のポイントをこの中に3〜5点。体言止め・句点なし。
- **要点や数値は太字**にするとマーカー表示になる。
:::

（目次はこの直後に自動挿入される。「この記事のポイント」とFAQの質問は目次から自動で除外される）

## 最初の見出し（h2・オレンジ帯になる）

本文。**強調したい箇所は太字にするとマーカー表示になる**（各段落1〜2箇所、要点・結論に）。

### 小見出し（h3・下線スタイル）

- 箇条書きも使える

:::point
重要なまとめはこのPOINT枠で囲む。1記事に1〜3個まで。
:::

> 引用はこの形式。出典を明記すること。

| 項目 | A | B |
|---|---|---|
| 比較表も | 使える | 使える |

## AI検索対策を伴走で進めるなら（featuring・PRパート）

見出しと導入文（2〜3文）だけ記事ごとにオリジナルで書く。その直後に下のタグを置くと、共通のPRパートが差し込まれる。

[blogparts:llmo-cta]

## よくある質問

### Q. 質問文（各QはH3・目次には出ない）
回答文。

### Q. 次の質問
回答文。
```

### front matter の項目

| 項目 | 必須 | 内容 |
|---|---|---|
| `title` | ✔ | 記事タイトル |
| `date` | ✔ | `YYYY-MM-DD` 形式 |
| `category` | ✔ | 下の既存カテゴリから選ぶ |
| `description` | ✔ | メタディスクリプション（100〜120字） |
| `image` | - | アイキャッチ画像パス。画像は `media/images/` に置く。省略時はプレースホルダ表示 |
| `author` | - | 省略時「岡村 希一」 |

### カテゴリ（勝手に増やさない）

- `AI検索対策`
- `ブランド戦略`
- `お知らせ`

新カテゴリが必要な場合は追加してよいが、乱立させない（最大5個目安）。

### 特殊記法（build.pyが変換する）

| 記法 | 変換結果 |
|---|---|
| `**太字**` | 蛍光マーカー強調 |
| `:::takeaways ... :::` | 「この記事のポイント」ボックス（冒頭。3〜5点） |
| `:::point ... :::` | POINT枠 |
| `[blogparts:llmo-cta]` | 共通PRパートを差し込む |
| `### Q. 質問` | FAQ見出し（目次には出ない） |

### PRパート（ブログパーツ）の運用

- 記事側で書くのは **featuringの見出し＋導入文（2〜3文）だけ**。その直後に `[blogparts:llmo-cta]` を置く。
- 差し込まれる固定PR本文は **`media/_template/blogparts/llmo-cta.html` の1ファイル**で管理。**ここを1回直せば、タグを埋め込んだ全記事に反映**される（記事mdは触らない）。
- PRの訴求を変えたい／新しいパーツを増やしたいときは、`media/_template/blogparts/` に `名前.html` を追加し、本文で `[blogparts:名前]` と呼ぶ。

## 執筆ルール（AEO/SEO・トンマナ）

- **結論ファースト**: 各h2の直後2〜3文で答えを述べる（AIに引用されやすくする）
- **問いを見出しに**: 読者が検索窓・AIに入力する言葉をそのままh2/h3に使う
- **一次情報を入れる**: nitoとしての見解・事例・データを必ず1つ以上入れる
- 文体: です・ます調。煽らない。「〜していきましょう！」の多用禁止
- 記事内リンク: 関連する既存記事があれば `/media/slug/` 形式で本文中からリンクする
- 「AI感」の出る表現を避ける: 「いかがでしたか」「本記事では〜について解説しました」等の定型句、過剰な箇条書き、絵文字
- **鍵括弧（「」）は「意味のある引用・専門用語・実際の質問文」にだけ使う**。「小手先の一手」「一つの信号」「やれば安心」のように、単語やフレーズを**強調したいだけ**で「」に入れるのはAI感が強く出るので禁止。強調したいなら太字（`**`）を使うか、そのまま地の文で書く。
  - OK例: 見出しで通説を引用する（`「〇〇すれば効果が出る」は本当か`）、実際の検索語・プロンプトを示す（`「〇〇に強い会社は？」`）、初出の専門用語・造語（`「Zapierパラドックス」`）、契約書等の用語を指す（`契約に「実行」が含まれるか`）
  - NG例: `こうした「小手先の一手」の多くは` → `こうした小手先の一手の多くは`
- **「理由は明確です。」「興味深いのは〜という点です。」「これが示すのは〜ということです。」のような、結論を先出しして解説する形の紋切り型の接続はAI感が出やすい**。地の文として自然に書き下す（例: `理由は明確です。Descriptは〜` → `Descriptは長年〜として知られており、〜`）。
- 海外事例・データを引用する記事は、**リサーチ結果の要約をそのまま貼らず**、他の段落と同じ文体・接続詞の呼吸で書き直す。事例セクションだけ急にAIっぽくなっていないか、書き終えたら記事全体を通しで読んで確認する。

## ボリュームと構成の推奨（AI引用を狙うための目安）

実際にAIに引用されている記事（ledge.ai・CINC等）は、8,000〜9,000字の網羅的な作りで、比較表や公式引用を複数持つ傾向がある。以下は**あくまで目安・推奨**であり、絶対条件ではない。**字数や表を稼ぐための水増しは逆効果**なので、読者が実際に気になる観点を足して自然に厚くすることを優先する。

- **推奨字数**: 5,000〜7,000字（網羅性を持たせる。ただし引き伸ばしはしない）
- **推奨h2数**: 6〜9個。各h2に、読者の疑問に答えるh3を1〜3個
- **各セクションの厚み**: 見出しだけ増やして中身が薄いのはNG。1つの見出しにつき、答え＋根拠＋具体例まで書く
- **比較表**: 自然に整理できる箇所があれば1〜2個入れる（SEOとの違い、施策の比較、類型の整理など）。表はAIが抜き出しやすい
- **権威ある引用（blockquote）**: Google公式ガイドや調査データなど、根拠として自然に引ける箇所があれば1〜2箇所。**日本語の一次ソースがあれば日本語で引く**（英語のまま貼らない）
- **一次情報**: nitoの実測・事例・見解を最低1つ。可能なら出典・期間・母数つきで
- **読者が気になる観点を足す**: 「この分野を調べた人なら次にこれも気になるはず」という論点を見出しとして補い、網羅性を上げる（水増しではなく観点の追加）
- **FAQ**: 5問前後。実際に投げられそうな質問を自然な言葉で
